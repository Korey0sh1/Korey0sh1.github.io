0x00：写在所有之前

0x01：BsidesCTF2021-shared-knote

tag：race condition 、userfaultfd、modprobe、Null pointer reference

题目分析

虽然在打开设备这边没有加锁，可以同时打开若干个该设备

但采用了引用计数器进行资源共享来防止UAF的利用，即只有第一次打开的时候才会申请object，之后再打开该设备只会使引用计数器—+1

至于释放该设备，就是只有当引用计数器=1时，才会free两个object，然后把一个指针清0

然后啊，鼠鼠我啊就8会惹，悲

漏洞利用

鼠鼠一直没有get到这题的考点是what

看了wp才略懂

假设我们开两个线程，一个线程open ，一个线程close，这样会发生什么呢

  1. 引用计数器 1——>0 (in close function)
  2. ##switch to open function
  3. 引用计数器 0——>1(in open function)
  4. kmlloc(0x10) (in open function)
  5. kmalloc(0x400) (in open function)
  6. ##switch to close function
  7. free() (in close function)
  8. free() (in close function)
  9. sknote_ptr = NULL (in close function)

在此种情况下，引用计数器不为0，但sknote_ptr已经是0了

那么问题来了，这样做能干什么捏

神奇の空指针解引用

一般来说有个常识，那就是对一个空指针进行解引用会直接dump

but在本题中，可申请内存页的最小地址是0，一般默认值是0x1000（敲霖娘谁一般看这个啊！！！)

所以说我们可以申请到0地址的page，在这种情况下对地址的0的解引用也就不成什么问题了

巧用userfaultfd条件竞争

虽然userfaultfd在5.11时候非root权限用不了，但是本题为了减少难度还是在非root用户开启了这个功能

于是可以先mmap地址为0的page，对此page注册userfaultfd，并设立一个flag标志

当对0地址进行解引用时，触发userfaultfd处理函数，更改flag的值，停止race condition

最后就用cpu_entry_area mapping这个未随机化的区域泄露kernel base，打个modprobe完工
